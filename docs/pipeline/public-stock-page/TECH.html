2 · Backend & infrastructure — architecture overview

Two halves, one direction of data flow:

  • Mac-side publisher (src/publisher.js) — subscribes to the watcher's 'family' event (the only event that fires in production; 'cycle' is emitted only by pollAllOnce(), which start() never calls), builds a privacy-filtered snapshot, and fire-and-forgets it to the public server. Disabled unless configured; can never affect watching or alerting (PRD D1, G4).
  • Public board server (board-server/, new top-level directory) — a single node:http file plus static board assets. Holds the latest snapshot in memory, mirrors it to one atomically-written JSON file for restart survival, and serves three public GETs plus the one token-gated POST. No database, no SSE, no Telegram, no other mutating endpoint (PRD D3–D5).

3 · Mac-side publisher — src/publisher.js

Responsibilities & wiring

  • Event source: the composed broadcast in src/index.js. Today index.js hands server.broadcast to the scheduler; with push configured it hands a two-line composition instead — server.broadcast(ev, payload) first (panel latency unaffected), then publisher.onEvent(ev). The publisher reacts only to 'family'; all other events are ignored. Zero changes to watcher.js or scheduler.js.
  • Debounce: a trailing 3 s timer (within the agreed 2–5 s band), reset by every 'family' event, unref()'d so it never holds the process open. Steady-state the 6 families land ~10 s apart (60 s cadence ÷ 6, staggered), so in practice this yields roughly one small push per family poll; the debounce exists to coalesce bursts — process-start catch-up, backoff recovery, wake-from-sleep — into a single push.
  • In-flight skip: if a push is still awaiting its response when the timer fires, the fire is skipped outright. No queue, no retry loop — the next family poll (≤ one cadence away) re-arms the debounce and carries strictly newer data. Combined with the fetch timeout this bounds the publisher to at most one outstanding request, ever.
  • Timeout: AbortSignal.timeout(10_000) on the fetch. This is deliberate hardening over telegram.js, whose postOnce has no timeout — here a hung socket would wedge the in-flight flag and silently stop all future pushes, so the abort is load-bearing, not cosmetic.
  • Live watchlist: the payload reads the panel server's .watchlist getter, never a boot-time reference — after POST /api/watchlist/remove (or a manual edit + reload) the very next push reflects the new family set. A boot-time ref would re-publish removed plans forever.
  • Shutdown: publisher.stop() clears the timer and aborts any in-flight fetch; index.js calls it in the existing graceful-shutdown path before server.stop().

Payload & privacy filter

  • Envelope: { v: 1, pushedAt: now(), state } — pushedAt is the Mac clock at build time; the server independently stamps receivedAt on accept (that, not pushedAt, drives public freshness).
  • State: the panel's own buildState({ watchlist: live, store }) — zero new state modeling (PRD D2) — with three subtractions: buildState is called with its default empty alarmed set so every plan carries alarm:false; the watcher key (uptime/Chrome/backoff lamp — operator internals) is deleted from the copy before wrapping; and each plan's lastCheckMs is stripped — per-plan poll timing is operator detail (DESIGN §8) and the board never renders it. Nothing else is rewritten.
  • Size: today's 37-plan snapshot serializes to ~12 KB — roughly 5× headroom under the server's 64 KB body cap.

New in src/state.js: outSinceMs + provider

planView() gains one symmetric field: outSinceMs = status === 'out' ? row?.last_change ?? null : null, exactly mirroring the existing inSinceMs. Plans seeded OUT that have never transitioned have last_change = null, so the field is null and the board renders nothing (the DESIGN's render-if-present contract). The generation view additionally passes through the family's provider field (whmcs for the HNL/VDS families) so the board's DC badge is data-driven; when absent the board defaults to DMIT defensively. Both fields also appear in the local panel's /api/state — additive and ignored by the existing panel JS.

Bearer auth — constant-time, fail-closed

  • At boot the server hashes the configured token once: sha256(PUSH_TOKEN). Per request it hashes the presented credential and compares digests with crypto.timingSafeEqual — hashing first guarantees equal-length buffers (timingSafeEqual throws on length mismatch) and leaks neither content nor length.
  • Fail-closed: if PUSH_TOKEN is unset or empty at boot, the server logs one loud warning and answers every push 401. The read side keeps serving whatever the snapshot file held — a misdeployed env file degrades to staleness, never to an open write endpoint.
  • The token value never appears in any log line or error body; 401 responses are uniform ({error:"unauthorized"}) regardless of which check failed.
  • Deliberately NOT ported from src/server.js: the loopback Host guard (this server legitimately answers a public hostname behind Caddy) and the CSRF Origin/Sec-Fetch-Site guard — there are no cookies and no browser-initiated writes here, so CSRF has no surface; the bearer token is the entire write-auth story, and an Origin-style guard could only ever reject legitimate pushes. What IS ported: the sendJson idiom, the SECURITY_HEADERS block (nosniff, frame DENY, no-referrer, CSP), the router-level try/catch that turns every handler rejection into an HTTP status (a request must never crash the process), and the body-cap/timeout reader.

5 · Push flow end-to-end

  • Real-world cadence: with 6 families at ~60 s cadence staggered ~10 s apart, a push lands ~3 s after each family poll — roughly every 10 s, ~12 KB each. Well inside PRD G2 (public within ~2 poll cycles) and negligible bandwidth; bursts coalesce via the debounce.
  • Warming up (fresh deploy, no file yet): /api/state returns state:null and the board renders its designed “warming up” empty state — the first push (≤ one cadence after the Mac side is configured) fills it.

Deploy runbook

  1. Pick a free loopback port on the box: check listening sockets and the other tenants' compose files; record the choice in the env file.
  2. Stage the app: create /opt/<app>/{app,data}; copy board-server/ into app/ and ops/compose.yml alongside; create the env file from .env.example with a fresh openssl rand -hex 32 token; chmod 600 it.
  3. Start + verify locally: docker compose up -d; from the box, curl 127.0.0.1:<host-port>/healthz (expect ok:true) and /api/state (expect the warming-up shape); confirm the healthcheck goes healthy.
  4. Wire the edge: copy the snippet (with the literal port) into the edge's snippet directory; add the one import line to the main Caddyfile if the import isn't glob-based.
  5. Restart the Caddy edge container — restart, not reload (house rule: reload can pin a stale config inode).
  6. Smoke-test every tenant domain on the box (HTTP status of each existing vhost) plus the new one: https://vps-stock.daichenlab.com/healthz over TLS proves cert issuance worked.
  7. Configure the Mac: add the two push keys to ~/.dmit-watch/config; restart the launchd watcher; watch the log for the “push publisher enabled” line and the first successful push; confirm the public board leaves “warming up” within one poll cycle.
  8. Prove staleness honesty (Gate-3 check): stop the watcher → banner appears after 5 min; start it → banner clears within ~30 s. Then add /healthz to the uptime monitor.

7 · Operational notes

TopicPolicy
Log hygieneNeither side ever logs the token, the Authorization header, or a request body. Mac: throttled failure lines + one recovery line. Server: one line per accepted push (family counts only); rejected pushes log status + reason class, with 401 floods coalesced into a per-minute counter line so an internet scanner can't flood the log.
Abuse surface & limitsPublic surface is two tiny GET JSONs + static files behind Caddy; the only expensive-ish path is token-gated and body-capped (64 KB, 10 s read timeout — same numbers as src/http-body.js). No rate limiter in v1: a 401 costs one hash + compare. If abuse ever shows up, rate limiting belongs at the Caddy edge, not in the app.
Mac offline = honest stalenessBy design the server keeps serving the last snapshot with receivedAt frozen; the board escalates amber → stale banner (PRD G3). /healthz stays 200 (the service is up); the monitor can alert separately on ageMs if the operator wants a “watcher dark > N min” signal distinct from “site down”.
Uptime monitoringPoint the operator's existing uptime monitor at GET /healthz: condition 1 = HTTP 200 (paging), optional condition 2 = ageMs < 600000 (non-paging staleness signal). No auth needed; the endpoint reveals nothing beyond what /api/state already exposes.
Token rotationGenerate a new token → update the box env file + docker compose up -d (recreates the container) → update ~/.dmit-watch/config → restart the watcher. Doing the server first means the gap is a few 401'd pushes (logged, harmless — next poll retries); no downtime for readers.
ClocksAll freshness math uses the server's clock (receivedAt, now); the Mac's pushedAt is informational only, so Mac↔server clock skew cannot fake freshness or staleness. Visitor clock skew is neutralized by the now field on /api/state.
TestabilityPublisher: injectable fetch/now/logger + exposed counters (the telegram.js pattern) — unit tests assert debounce, skip, privacy filter, and throttled logging with a fake clock. Server: factory takes {port: 0, snapshotFile: tmpdir, token, now} — node --test integration covers the full guard chain (401/400/408/413/422/200), atomic-write ordering, corrupt-file boot, and the no-store header, matching the repo's existing test style.
Invariant recap (audit surface, PRD G4/G6). The public server contains no Telegram code path, no SQLite, no SSE, and exactly one mutating route, which is bearer-gated and fail-closed. The Mac's watcher/alerting behavior is bit-identical with the publisher disabled, and the publisher is disabled unless both push keys are present. Secrets exist only in the box env file and ~/.dmit-watch/config; the repo ships .example files only.

8 · Frontend · Client architecture

The public board is a separate entry — its own HTML, its own JS modules, its own CSS file — living under the vps-stock server's own static root (board-server/public/). Nothing from the panel's public/ directory is ever mounted or imported, so alarm/audio/POST code is excluded by construction, not by runtime flags (DESIGN §8/§10). The client is framework-free ES modules: one fetch loop, pure HTML-string builders, one paint per new snapshot.

10 · Frontend · Polling, freshness & the age ticker

Two independent clocks: a fetch loop that refreshes data, and a 1-second ticker that keeps the age label honest between fetches. Freshness is computed from receivedAt only, against the server-corrected clock (the envelope's now) — a live tab holding old data must read as old (DESIGN §5); fetch success is never a freshness signal.

Timing constants (single source in board.js)

ConstantValueMeaning
POLL_MS30 000Base fetch interval (orchestrator default #2)
POLL_JITTER_MS0–5 000 random per cycleDe-synchronizes many open tabs so pushes don't produce request bursts
TICK_MS1 000Age-label ticker
AGING_MS120 000Fresh → aging threshold
STALE_MS300 000Aging → stale threshold (PRD G3)
BACKOFF_CAP_MS300 000Failure interval = POLL_MS × 2ⁿ capped here; reset to 0 failures on any success
  • Re-render gate. A fetch that returns the same receivedAt repaints nothing — the DOM is only rebuilt when a genuinely newer snapshot arrived. Even though a push lands roughly every family poll (~10 s steady-state), the board polls every 30 s and each poll picks up at most one newer snapshot — so the steady state costs at most two paints per minute.
  • FLIP promotion. Around each real re-render, board.js captures getBoundingClientRect() for every [data-id] card, repaints, then plays the invert transform exactly as the panel's renderPanel(true) does — a restocked card visibly travels from the wall to the hero. Skipped entirely under prefers-reduced-motion.
  • Tab hidden. visibilitychange → hidden clears both timers (no background fetching); → visible fetches immediately, recomputes the tier once (the board may have gone stale while hidden), and restarts both timers.
  • Clock skew. Each successful fetch records skewMs = Date.now() − body.now (0 when now is absent — render-if-present); the ticker computes age as (Date.now() − skewMs) − receivedAt, clamped ≥ 0. A visitor's wrong clock can neither fake freshness nor raise a false stale banner.

12 · Frontend · Accessibility & performance

Accessibility (DESIGN §7 made concrete)

  • Semantics — header · nav aria-label="Jump to datacenter" · section aria-labelledby for IN/waiting zones · DC groups as section aria-label="{city}" · footer. Status pills are words, never color-only.
  • Live regions without chatter — the freshness pill's visible ticking label is aria-hidden="true"; a visually-hidden sibling with role="status" aria-live="polite" is updated only on tier transitions (“Board data is fresh — updated under 2 minutes ago” / aging / stale copy). This implements the design's “the 1 s tick must not re-announce” requirement literally: screen readers hear three transitions, not 86 400 ticks. The stale banner keeps its own polite role="status".
  • Keyboard — tab order: jump chips → Buy links → footer links. Cards are not tabbable (no tabindex, unlike the panel); the Buy anchor is each IN card's only interactive element. Shared :focus-visible outline (2px blue, offset 2px) on chips, Buy, and footer links. Anchor targets carry scroll-margin-top clearing the sticky zone.
  • Reduced motion — one media block disables: FLIP promotion, card hover lift, stale-pill pulse, html scroll-behavior:smooth. All states remain fully legible via color + copy.
  • Contrast — token pairs are the panel's proven ≥ 4.5:1 set; the dimmed stale board is intentionally sub-contrast while the full-contrast banner carries the truth.
  • Touch — Buy buttons and jump chips ≥ 40px effective hit area (padding included); Buy opens a new tab with rel="noopener".

9 · Defects & observations

D1 · LOW · .gitignore does not cover the real deploy env file's actual name.
ops/vps-stock-compose.yml loads env_file: ./vps-stock.env, and vps-stock.env.example instructs “copy to vps-stock.env beside the compose file on the box”. If the box's deploy directory is a git clone (a common pattern in this fleet), the real token-bearing file sits in the working tree and no ignore pattern matches it — .env and .env.* match neither vps-stock.env nor ops/vps-stock.env. Repro: git check-ignore ops/vps-stock.env → exit 1 (not ignored). An accidental git add -A && git push on the box would publish the push token to the PUBLIC repo. Suggested hardening (one line, for the implementer — QA changed nothing): add *.env with !*.env.example, or the literal ops/vps-stock.env, to .gitignore.

No functional, security, or rendering defects found. All observed behavior matched the TECH contract exactly, including the deliberate design points (bearer-only write auth with no CSRF port, restart honesty, fail-closed empty token, 503 placeholder path untested here because the bundle ships in this branch).

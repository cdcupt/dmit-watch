2 · Problem & context

dmit-watch already answers "is a DMIT Premium plan buyable right now?" — but only on the operator's Mac (localhost:7331). The stock question is interesting to other people (friends, forum/VPS-community members), and today the only way to share it is a screenshot that is stale the moment it's taken.

Why we can't "just move the website": the watcher's only trustworthy stock source is DMIT's Cloudflare-protected cart.php. Recon (v3) and production experience prove only a real, locally-running Chrome passes the challenge; reads from datacenter IPs (bwh-prod) get 403 Just a moment. The reader must stay on the Mac. What can move is the display: the Mac already computes a complete state snapshot every poll — it can push that to a public host.

4 · Goals & success criteria

  • G1 — Live public board. https://vps-stock.daichenlab.com serves the stock board over HTTPS; all watched plans (37 today) appear with status, price, and buy deep link. Criterion: a cold visitor sees current stock without any auth.
  • G2 — Freshness within one poll cycle. A status change on the Mac panel is visible publicly within ~2 poll cycles (≤ ~3 min worst case). Criterion: flip observed locally → public JSON updated on next push.
  • G3 — Honest staleness. The page always shows "updated Xs ago"; if the last push is older than 5 min it shows a clearly visible data may be stale — watcher offline banner. Criterion: stop the Mac watcher → banner appears; restart → banner clears.
  • G4 — Read-only & quiet. No alarm sound, no browser notifications, no auto-buy, no Telegram from the server, no mutating endpoints exposed. Criterion: audit of the public surface finds GETs only (+ the one authenticated push POST).
  • G5 — Multi-tenant-safe deploy. Own container + own Caddy snippet on bwh-prod; zero disruption to existing tenants. Criterion: all existing daichenlab.com sites still answer after deploy.
  • G6 — No secrets in the public repo. Push token lives only in ~/.dmit-watch/config (Mac) and the server's env file. Criterion: secret scan on the PR is clean.

5 · Scope

In

  • New push publisher in the Mac watcher: after each poll cycle, POST the current state snapshot to the public service (fire-and-forget, never blocks or breaks watching; disabled unless configured).
  • New tiny vps-stock service (this repo, server/ or similar): authenticated POST /api/push, public GET /api/state, serves the static board page; keeps the latest snapshot in memory + a file so restarts don't blank the page.
  • New public board page: the existing panel's visual language, trimmed to public-safe content — grouped by datacenter/generation, plan cards with IN/OUT/UNKNOWN→shown as "checking", price, key specs (as available in the snapshot), buy link, "in stock since / out since" times, page-level freshness + stale banner. Mobile-friendly (320/390/1440). No alarm, no audio, no notification prompts.
  • Deploy to bwh-prod per house rules: own container on a free localhost port, own Caddy snippet + one import line, docker restart 9relay-caddy, smoke-test all tenant domains.

Out (explicitly)

  • Moving the watcher/Chrome off the Mac (impossible: Cloudflare).
  • Server-side alerting of any kind (Telegram, email, push) — the server only displays.
  • Operator internals on the public page: watcher-health tab, backoff/blind details, Telegram log, silence/remove controls.
  • Historical charts / transition history page (the snapshot's per-plan "since" timestamps are enough for v1).
  • Accounts, rate-limited APIs for third parties, embeds — v1 is a human-readable page + one JSON GET.

6 · Key decisions & constraints

#DecisionRationale
D1Push, not pull. The Mac POSTs snapshots to bwh-prod; the server never contacts the Mac or DMIT.The Mac isn't publicly reachable; DMIT is Cloudflare-walled from datacenters. Push is the only direction that works.
D2Reuse the existing snapshot shape. The push body is the local panel's /api/state JSON (plus a schema version + pushed-at timestamp).Zero new state modeling; the board renders what the panel already renders; the local panel stays the source of truth.
D3Bearer-token auth on push only. Long random token; config on both ends; pushes without it are rejected 401.Only the operator's Mac may write; everyone may read.
D4No database. Latest snapshot in memory, mirrored to one JSON file for restart survival.It's a cache of one document; a DB adds operational surface for nothing.
D5Public page is quiet. No alarm banner/audio/Notification API; UNKNOWN renders as a neutral "checking…" state, never as scary red.PM decision (2026-07-04). Public visitors get information, not alarms; alerting stays the operator's private Telegram.
D6Same public repo. Server + page live in cdcupt/dmit-watch.One project, one paper trail; the repo is already the public home of the tool. Secrets stay in env/config files.

7 · Push contract (sketch — final in TECH doc)

Aspectv1
EndpointPOST https://vps-stock.daichenlab.com/api/push
AuthAuthorization: Bearer <token> — token in ~/.dmit-watch/config (Mac) and server env; 401 otherwise
Body{ v: 1, pushedAt, state: <local /api/state JSON> }
CadenceAfter each completed poll cycle (~60 s per family, jittered) with light debounce; retry is "next cycle" (no queue)
Failure modePush errors are logged locally and NEVER affect watching/alerting; the public page goes stale-honest (G3)
Public readsGET / (board page) · GET /api/state (latest snapshot + receivedAt)

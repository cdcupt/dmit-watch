1 · ★ At a glance

2 · Problem & context

dmit-watch already answers "is a DMIT Premium plan buyable right now?" — but only on the operator's private panel (localhost:7331). The stock question is interesting to other people (friends, forum/VPS-community members), and today the only way to share it is a screenshot that is stale the moment it's taken.

Why the watcher needs a RESIDENTIAL IP: the watcher's only trustworthy stock source is DMIT's Cloudflare-protected cart.php. A feasibility spike (2026-07-04) tested both host classes with the production reader: from the prod host's datacenter IP every read stays walled ("Just a moment" on every pass); from a residential-IP VPS the operator owns, the first read's 403 challenge auto-cleared in 6.4 s with 5/5 plan names parsed, and the next read returned 200 in 0.9 s (cf_clearance persists in the Chrome profile). So the two-box split is required: the watcher (node + dedicated headless Chrome) runs on the residential VPS; the prod host only displays. The watcher already computes a complete state snapshot every poll — it pushes that to the public host.

4 · Goals & success criteria

  • G1 — Live public board. https://vps-stock.daichenlab.com serves the stock board over HTTPS; all watched plans (37 today) appear with status, price, and buy deep link. Criterion: a cold visitor sees current stock without any auth.
  • G2 — Freshness within one poll cycle. A status change on the watcher's panel is visible publicly within ~2 poll cycles (≤ ~3 min worst case). Criterion: flip observed locally → public JSON updated on next push.
  • G3 — Honest staleness. The page always shows "updated Xs ago"; if the last push is older than 5 min it shows a clearly visible data may be stale — watcher offline banner. Criterion: stop the VPS watcher → banner appears; restart → banner clears.
  • G4 — Read-only & quiet: no alerting anywhere in this deployment. No alarm sound, no browser notifications, no auto-buy, no mutating endpoints exposed — and no Telegram: the watcher runs with no Telegram credentials (no-op notifier; the feature stays optional for self-hosters). Criterion: audit of the public surface finds GETs only (+ the one authenticated push POST); the watcher boot log shows "telegram: disabled (no credentials)".
  • G5 — Multi-tenant-safe deploy. Own container + own Caddy snippet on the prod host; zero disruption to existing tenants. Criterion: all existing daichenlab.com sites still answer after deploy.
  • G6 — No secrets in the public repo. Push token lives only in ~/.dmit-watch/config (watcher host) and the server's env file. Criterion: secret scan on the PR is clean.
  • G7 — Nothing on the operator's Mac. The watcher (node + dedicated Chrome) runs entirely on the residential VPS; the Mac's launchd watcher is decommissioned at cutover. Criterion: after cutover the Mac has no dmit-watch launchd agent loaded and the public board stays fresh from the VPS alone.

5 · Scope

In

  • New push publisher in the watcher: after each poll cycle, POST the current state snapshot to the public service (fire-and-forget, never blocks or breaks watching; disabled unless configured).
  • Watcher deployment on the residential VPS: repo at /opt/dmit-watch, two systemd units (dedicated headless Chrome + the watcher under a restart policy — the same pair scripts/start.sh + launchd managed on the Mac), config holding only the push settings; the local panel (:7331) stays loopback-only (operator debugging via SSH tunnel).
  • Telegram-optional code change: absent Telegram credentials no longer abort boot — the watcher starts with a no-op notifier and logs one line ("telegram: disabled (no credentials)"); blind escalations and restock edges become panel-only. The feature stays available to self-hosters who do configure credentials.
  • Decommission runbook for the Mac launchd agent (run only after the VPS watcher is verified green).
  • New tiny vps-stock service (this repo, server/ or similar): authenticated POST /api/push, public GET /api/state, serves the static board page; keeps the latest snapshot in memory + a file so restarts don't blank the page.
  • New public board page: the existing panel's visual language, trimmed to public-safe content — grouped by datacenter/generation, plan cards with IN/OUT/UNKNOWN→shown as "checking", price, key specs (as available in the snapshot), buy link, "in stock since / out since" times, page-level freshness + stale banner. Mobile-friendly (320/390/1440). No alarm, no audio, no notification prompts.
  • Deploy to the prod host per house rules: own container on a free localhost port, own Caddy snippet + one import line, restart the shared Caddy edge, smoke-test all tenant domains.

Out (explicitly)

  • Telegram / alerting of any kind in our deployment — no credentials are configured anywhere; the code path remains an optional feature for self-hosting users of the public repo. No other alerting is added.
  • Server-side alerting of any kind (Telegram, email, push) — the server only displays.
  • Operator internals on the public page: watcher-health tab, backoff/blind details, Telegram log, silence/remove controls.
  • Historical charts / transition history page (the snapshot's per-plan "since" timestamps are enough for v1).
  • Accounts, rate-limited APIs for third parties, embeds — v1 is a human-readable page + one JSON GET.

6 · Key decisions & constraints

#DecisionRationale
D1Push, not pull. The watcher on the residential VPS POSTs snapshots to the prod host; the server never contacts the watcher host or DMIT.The watcher host exposes no public surface, and DMIT is Cloudflare-walled from datacenter IPs — pushes originate from the residential VPS; push is the only direction that works.
D2Reuse the existing snapshot shape. The push body is the local panel's /api/state JSON (plus a schema version + pushed-at timestamp).Zero new state modeling; the board renders what the panel already renders; the local panel stays the source of truth.
D3Bearer-token auth on push only. Long random token; config on both ends; pushes without it are rejected 401.Only the operator's watcher may write; everyone may read.
D4No database. Latest snapshot in memory, mirrored to one JSON file for restart survival.It's a cache of one document; a DB adds operational surface for nothing.
D5Public page is quiet. No alarm banner/audio/Notification API; UNKNOWN renders as a neutral "checking…" state, never as scary red.PM decision (2026-07-04). Public visitors get information, not alarms; there is no alerting anywhere in this deployment (see D8) — the website is the product.
D6Same public repo. Server + page live in cdcupt/dmit-watch.One project, one paper trail; the repo is already the public home of the tool. Secrets stay in env/config files.
D7The watcher hosts on a residential-IP VPS. Node + dedicated headless Chrome move to a residential-IP VPS the operator owns; nothing runs on the Mac.Validated by a feasibility spike with the production reader: on the residential IP the first read's 403 challenge auto-cleared in 6.4 s (5/5 plans parsed) and the next read was 200 in 0.9 s; the prod host's datacenter IP stays walled ("Just a moment" every pass). The two-box split is empirical, not preference.
D8Telegram optional-off. Our deployment configures no Telegram credentials; the watcher boots with a no-op notifier and one log line. The codebase keeps Telegram as an optional feature.The public OSS repo has self-hosting users who rely on Telegram; our deployment's product is the website. Blind escalations and restock edges become panel-only. No other alerting is added.

7 · Push contract (sketch — final in TECH doc)

Aspectv1
EndpointPOST https://vps-stock.daichenlab.com/api/push
AuthAuthorization: Bearer <token> — token in ~/.dmit-watch/config (watcher host) and server env; 401 otherwise
Body{ v: 1, pushedAt, state: <local /api/state JSON> }
CadenceAfter each completed poll cycle (~60 s per family, jittered) with light debounce; retry is "next cycle" (no queue)
Failure modePush errors are logged locally and NEVER affect watching/alerting; the public page goes stale-honest (G3)
Public readsGET / (board page) · GET /api/state (latest snapshot + receivedAt)

9 · Rollout

  • Branch feat/public-stock-page; artifacts under docs/pipeline/public-stock-page/; normal PR at publish.
  • DNS: vps-stock.daichenlab.com → the prod host's A record — already created by the PM (DNS-only).

3 · Live evidence

4 · Lens A — new-user & polish (SHIP)

Walking in cold from a shared link, this reads like a finished, trustworthy product within a few seconds. The header ("VPS · STOCK WATCH", "read-only board"), a live "updated 12s ago" pill, an info note explaining a watcher polls each provider's store every ~60s, and a "How this works / Not affiliated / open source" footer all mean I never had to guess what I was looking at. The single most important question — "what can I actually buy right now and for how much?" — is answered instantly: 15 filled-green "IN STOCK" cards with prices and Buy-now buttons sit at the top, and everything sold out is pushed below into a muted "WAITING FOR RESTOCK" section with red "OUT OF STOCK" pills and no button. The in/out distinction is unmistakable. The freshness pill genuinely ticks live (I watched it go 3s→9s), which makes the data feel real rather than a stale snapshot. Buy-now links open the correct DMIT/qq.pw cart in a new tab with proper rel=noopener. It's responsive with zero body overflow down to 320px. My only real friction: the Buy-now deep link lands on the provider's generation-level page rather than the exact plan/size on the card, and clicking a jump chip for an all-in-stock datacenter (HKG/TYO) drops you on an essentially empty waiting-section header.

5 · Lens B — abuse / security / integrity (SHIP)

As a distrustful beta tester I tried hard to break correctness and trust, and the board held up well. Page-to-API integrity is exact: the live page renders exactly the 15 IN-stock plans that GET /api/state reports (same names and prices — HKG.AS3 ×5, HKG.AN5 ×5, TYO.AS3 ×5), the "15 in stock / 22 waiting / 37 watched" strip matches counts.total/in/waiting, and per-datacenter LAX 16 · HKG 10 · TYO 7 · HNL 4 matches counts.byLoc. No plan is shown IN that the JSON says is out. The security surface is locked down: POST /api/push returns 401 with no token AND with a garbage Bearer token (auth is checked before any body work), POST /api/state → 405 (Allow: GET), GET /api/silence and GET /api/watchlist/remove → 404, unknown paths → 404, and every path-traversal attempt (/js/../../server.js, percent-encoded %2e%2e, /../package.json, /js/../../../etc/passwd) returns 404 {"error":"not found"} — never file contents. Cache-Control: no-store is present on both /api/state and /healthz. Extra hardening I confirmed: strict CSP (default-src 'self', object-src 'none', frame-ancestors 'none'), HSTS, X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy no-referrer, and NO Access-Control-Allow-Origin even when I forged Origin: https://evil.example (same-origin only, no CORS leak). There is zero alarm/audio/notification behavior — I grepped both index.html and js/board.js and found no Audio/AudioContext/Notification/requestPermission/vibrate/beep code at all. Freshness is honest: the pill read "updated 3–4s ago" while /healthz reported ageMs≈383, and it ticks every second. Weird inputs are handled: ?demo=1 renders a labeled "demo data" chip with clearly different sample data (2 in stock, neutral CHECKING… cards, spec rows) and does NOT corrupt the real page — the server-side /api/state?demo=1 still returned the true 15-in payload, proving demo is client-only; garbage query params (?'"<script>=1&foo[]=bar) are ignored and return 200. Responsive down to 320px with zero horizontal overflow at 1440/390/320.

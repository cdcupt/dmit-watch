5 · Lens B — abuse / security / integrity (SHIP)

LENS B (abuse/security/integrity) black-box pass against the LIVE https://vps-stock.daichenlab.com subscription surface. Endpoints discovered from served js/subscribe.js (not source): POST /api/subscribe {token,chatId,planIds}, /api/subscription/lookup|update|delete, /api/chatid {token}. Token regex ^\d{8,12}:[A-Za-z0-9_-]{30,}$, chat ^-?\d+$. CHARTER RESULTS — all PASS except two LOW notes: (1) Subscribe junk-but-well-formed token (1234567890:AAA…33) -> 422 {"reason":"token_rejected"}, ~3.3s (real Telegram getMe round-trip), token NOT echoed. NOTHING stored: same-pair lookup -> 404 {"error":"not found"}. Used 1 real junk-subscribe attempt; a 2nd was consumed capturing 429 headers (2/2 cap respected). (2) Malformed bodies -> 400: invalid JSON {"error":"invalid JSON body"}, wrong content-type -> 400, empty -> 400 {"error":"invalid request"}. Oversized 9KB -> 413 {"error":"request body too large"}. (3) Anti-oracle: 3 unknown well-formed pairs (incl. negative group id) all return byte-identical 404 {"error":"not found"}, timing 0.93/0.96/1.09s — no shape or timing signal to distinguish which of token/chatId is wrong. Malformed-shape pair -> 400 (distinct, expected). No oracle. (4) POST /api/chatid junk token -> 422 {"reason":"token_rejected"}, no token echoed, clean failure (2/3 calls used). Rate-limit design CONFIRMED on subscribe: 7 rapid POSTs -> 429 {"error":"rate limited"} with retry-after: 3483 (~58min hourly window). chatid sits in a separate bucket (still 422 after subscribe was 429'd). No token value echoed in any error/limit response. (5) GET /api/state 200, Cache-Control: no-store, cadenceSec: 300 present; /healthz 200 {ok,receivedAt,ageMs}; page 200 public. Board GET + page stayed available (200) even while subscribe was rate-limited — read path correctly isolated. (6) Round-1 regression intact: POST /api/push no-token AND garbage-token -> 401; POST /api/state -> 405; traversal /js/../../etc/passwd, %2e%2e encoded, /../server.js -> 404; unknown paths -> 404. Security headers unchanged on all responses: CSP (default-src 'self'; object-src 'none'; frame-ancestors 'none'; script/style/connect-src 'self'; form-action 'none'), HSTS max-age=31536000; includeSubDomains, X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy no-referrer. No CORS leak: forged Origin on /api/state returns no Access-Control-Allow-Origin; OPTIONS /api/subscribe -> 405 allow: POST (no CORS). (7) Storage clean (CDP visit): localStorage {}, sessionStorage {}, document.cookie "", context cookies []. Panel never persists token/chatId; no credential storage anywhere. (8) Integrity: rendered stats (15 in stock / 22 waiting / 37 watched) match GET /api/state counts exactly (in 15, waiting 22, total 37). Cadence copy present ("5 minutes"), no "~60s" remnants; restock-alert bell pill present.
SevAreaFinding
LOWroute contract / info-disclosurePOST /api/silence and POST /api/watchlist/remove return 405 (allow: GET) rather than the round-1-documented 404 (F26/charter). Verified benign: it is a generic static-catch-all — GET on ANY unknown path -> 404 {"error":"not found"}, POST on ANY unknown path (e.g. /totally-random-path, /api/zzz-random) -> 405 allow:GET. No mutating silence/watchlist route exists, no data leaked, no mutation performed. Pure doc-vs-behavior mismatch; not a security hole.
LOWrate-limit / shared-NAT availabilitySubscribe rate limit is IP-keyed with a ~1h window and low threshold (~6/h). On trip it returns 429 {"error":"rate limited"} retry-after: 3483 (~58 min). Users behind shared NAT/CGNAT/corporate proxy could collaterally lock each other out of the subscribe/chatid flow for up to an hour on one IP. Mitigation posture is otherwise correct (429 + Retry-After per F51, no token echo, board GET/page stay fully available). Consider IP+token composite keying or a shorter window to reduce collateral lockout.

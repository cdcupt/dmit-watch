1 · ★ At a glance

2 · Problem & context

The board answers "what's in stock right now?" — but only if you happen to be looking. The operator no longer receives restock alerts (Telegram was disabled in round 1 by design), and visitors have no way to be told when the plan they want comes back. Meanwhile, polling providers every ~60 s is more aggressive than the product needs.

Round 2 closes both: self-serve, per-plan Telegram alerts for anyone, and a gentler ~5-minute poll cadence. Notification logic lives server-side in the board-server — it already receives every snapshot, so it can detect OUT→IN transitions by diffing; the watcher needs no new responsibilities.

3 · Goals & success criteria

  • R1 — 5-minute provider polling. The watcher polls each provider family every ~5 min (config change). Criterion: watcher logs show ~300 s family cadence; DMIT sees ≤ 1/5th of today's requests.
  • R2 — Honest freshness at the new cadence. The page's freshness ladder and copy are recalibrated so normal 5-min operation never shows amber/stale, while a genuinely dead watcher still surfaces prominently. Criterion: 24 h of normal operation with zero false stale banners; stopping the watcher still raises the banner.
  • R3 — Per-plan subscribe. A visitor can pick exact plans, provide their own bot token + chat id, and get a confirmation message on subscribe (which also proves the token/chat work). Criterion: end-to-end subscribe → confirmation card arrives in Telegram.
  • R4 — One digest card per restock event. When subscribed plans flip OUT→IN, the subscriber receives ONE message listing all their newly-IN plans (name, price, datacenter, buy link) — batched across families within a short window; a plan re-alerts only after it has gone OUT again (armed semantics). Criterion: a multi-plan restock produces exactly one message per subscriber.
  • R5 — Manage & unsubscribe. Subscribers can view/update their plan list and unsubscribe, authenticated by the same token + chat id pair. Criterion: full lifecycle works from the page.
  • R6 — Abuse-safe & private. Subscribe endpoints are rate-limited and validated; tokens are stored server-side with restrictive permissions, never logged, never echoed to any public endpoint, never in the repo. Failing tokens (revoked bots) auto-disable after repeated failures. Criterion: QA's abuse suite passes; a revoked token stops being retried.
  • R7 — UNKNOWN never notifies. Only a confirmed IN triggers a digest (consistent with the detector's false-IN-impossible doctrine). Criterion: test proves UNKNOWN/checking transitions send nothing.

4 · Scope

In

  • Watchlist cadence change (~60 s → ~300 s) + every knock-on recalibration (freshness ladder, page copy, blind-net timing review).
  • Board-server: snapshot diffing edge detector with per-plan armed state (persisted); subscriber store (embedded, zero-ops — e.g. node:sqlite on the existing data volume); Telegram sender using each subscriber's own token; digest batching window; subscribe/verify/manage/unsubscribe API; rate limiting.
  • Board page: subscribe UI — per-plan selection, token + chat id form with clear BotFather instructions, manage/unsubscribe flow, success/error states. No accounts, no cookies beyond what the flow strictly needs.
  • Docs/tests per house standard; README + setup-skill updates at publish.

Out (explicitly)

  • A shared/official service bot (PM chose BYO-token; may revisit later).
  • Non-Telegram channels (email, webhooks, RSS).
  • Operator alerting on the watcher host (stays disabled — the server notifies subscribers only).
  • Price-drop / spec-change notifications (restock only).
  • Watcher-side changes beyond the cadence number.

5 · Key decisions & constraints

#DecisionRationale
D1Server-side edge detection by snapshot diffing. The board-server compares each accepted push against the previous per-plan status with armed/disarm semantics mirroring the watcher's own edge engine.The server already sees every snapshot; the watcher contract stays frozen; subscribers' logic lives where the subscribers live.
D2BYO bot token + chat id is both the delivery credential and the manage/unsubscribe identity.PM decision. Zero shared-bot ops; abuse-limited by construction (a token can only message chats that already talked to that bot — effectively only the subscriber themself).
D3Confirmation message on subscribe. The server sends a test card immediately; a failed send rejects the subscription.Proves token + chat id validity at the door; no dead subscriptions.
D4Digest batching window. Edges collect for a short window (~60–90 s, tech doc finalizes) after the first edge before sending, so a mass restock staggered across family polls still yields ONE card.PM requirement: one readable card, never a ping-storm. Family polls are staggered; without a window a 7-family restock could mean up to 7 messages.
D5Armed semantics persisted server-side. A plan that alerted stays disarmed until observed OUT again; state survives server restarts.No duplicate alerts across restarts; mirrors the proven watcher behavior.
D6Embedded storage on the existing volume (node:sqlite), no external DB, no new npm deps.Keeps the board-server's zero-dep, copy-one-directory deployability.
D7Freshness ladder derives from the poll cadence (served to the page in the API envelope) instead of hard-coded 2/5-min thresholds.One knob: a future cadence change can't silently break staleness honesty again.
D8Round-1 invariant repeal, scoped: the board-server MAY reach api.telegram.org and nothing else; the "GET-only public surface" audit becomes "the only mutating routes are /api/push (bearer) + the subscription routes (rate-limited, validated)".The invariant's intent (no operator alerting, no surprise egress) survives; the tests get updated deliberately rather than deleted.

6 · The digest card (format sketch — final in DESIGN/TECH)

🟢 Restock — 3 of your plans are IN STOCK (vps-stock.daichenlab.com)

• HKG.AS3.Pro.TINY — $39.90/mo — Hong Kong · Buy now
• HKG.AN5.Pro.MINI — $149.90/mo — Hong Kong · Buy now
• TYO.AS3.Pro.STARTER — $45.90/mo — Tokyo · Buy now

as of 14:32 UTC · data can lag ~5 min · manage subscription: vps-stock.daichenlab.com

  • One message per subscriber per event window; only THEIR subscribed plans appear.
  • Each line: plan · price · datacenter · direct buy link. Header counts; footer carries the honesty note + manage pointer.
  • Confirmation card on subscribe uses the same visual language ("You're subscribed to N plans: …").

7 · Risks & mitigations

RiskImpactMitigation
5-min cadence misses ultra-short restocksSubscriber loses a buying windowAccepted by PM (①). The digest footer states the lag honestly.
Storing third-party bot tokensCredential exposure if the host is compromisedTokens live only in the store on the data volume (0600-class perms), never logged/echoed; docs tell users the token can be revoked in one tap via @BotFather; auto-disable on repeated send failures.
Subscription endpoint abuse (spam signups, token stuffing)Server load; Telegram probingPer-IP rate limits + body caps; confirmation-send acts as a validity gate; store size cap with clear 429/413 errors.
Telegram API slow/down at notify timeDelayed/missed digestsBounded retries with backoff inside the event window; failures logged + counted per subscriber; armed state only cleared on confirmed send... (tech doc finalizes exact semantics so a failed send does not permanently swallow an alert).
Ladder recalibration wrong → false "stale" banners at 5-min cadenceTrust erosionD7 (cadence-derived thresholds) + a 24 h soak criterion (R2) before Gate 3 sign-off.
Mass restock = burst of sends across many subscribersRate limits on user bots (per-bot limits are per-token — fine); server burstEach subscriber has their own token (independent limits); sends are queued with modest concurrency.

8 · Rollout

  • Branch feat/subscriptions; artifacts under docs/pipeline/subscriptions/; same gate discipline (Design → Gate 1, Tech → Gate 2, build/QA/beta → Gate 3).
  • Cadence config + ladder recalibration can ship in the same PR; the watcher host picks up the cadence via git pull + service restart.
  • Deploy = update /opt/vps-stock/app on the prod host (data volume untouched; new store file appears beside the snapshot) + watcher restart on the residential VPS.
  • R2's 24 h soak runs between deploy and final sign-off; the beta panel exercises subscribe/manage/digest with a real disposable bot token.

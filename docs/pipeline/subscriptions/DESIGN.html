2 · Entry points — placement & rationale

EntryWhereBehaviorWhy here
A · Global CTARight end of the .stats strip: pill 🔔 Get restock alerts (blue-soft bg, blue-ink text — the page's only blue pill, so it reads as an action, not a status)Opens panel → Subscribe tab, nothing pre-checkedVisible without scrolling on every viewport; lives in the stats strip so the header keeps its round-1 “exactly one lamp” contract — no repeal needed
B · Per-card bellFooter of every OUT and Checking… card: ghost button 🔔 Notify me (dashed border, mono, accent ink). IN cards get no bellOpens panel → Subscribe tab with that plan pre-checked, its DC group expanded and scrolled into view; tray shows “1 selected”Intent is highest exactly where disappointment happens — the OUT card of the plan you wanted. Fills the footer dead space render.js already leaves on OUT cards. IN cards keep Buy now as their single competing-free action (a bell next to Buy hurts the page's primary conversion)
C · Deep link/?manage=1 — printed in the footer of both Telegram cardsOn load: opens panel → Manage tab. No secrets ever ride the URLThe Telegram card is the only place a subscriber reliably returns to; one tap gets them back to edit/unsubscribe

Rejected alternatives: bell on IN cards (competes with Buy); CTA in the header (would repeal the one-lamp contract for no gain); floating action button (covers cards at 320 px and collides with the dev demo chip). A search box in the picker was also rejected — 22–37 plans in 4 collapsible groups with per-group select-all is faster than typing on mobile; revisit if the watchlist passes ~80 plans.

Plans that are currently IN are still selectable in the picker (a subscription is about the next restock). The picker's status column plus one helper line carry the honesty: “Plans in stock now alert on their next restock — after they've gone out and come back.”

Step 2 — the BYO-token form, field by field

ElementSpec
InstructionsNumbered 3-step list. Step 1 links https://t.me/BotFather (new tab, rel="noopener") and shows /newbot as a copy-chip (mono code + a Copy button, navigator.clipboard, label flips to “Copied ✓” for 1.5 s). Step 2 is the press-Start gate in bold — it is the #1 drop-off and precondition for both chat-id discovery and delivery. Step 3 points at the fields below.
Token fieldtype="text" (visible — users verify by eye; it's pasted, not memorized), autocomplete="off" spellcheck="false", mono font. Client format check on blur: /^\d{8,10}:[A-Za-z0-9_-]{35}$/. Never prefilled after a successful submit; never persisted to localStorage (a credential is not a convenience).
Chat id fieldinputmode="numeric", format check /^-?\d+$/. Beside it: Find my chat id button (enabled once the token passes format) → POST /api/chatid — the server calls Telegram getUpdates with the pasted token and fills the field on success. Failure copy tells the user to press Start first. Manual entry always works as the fallback; a “what's a chat id?” disclosure (<details>) explains it in two sentences.
Inline errorsMono, --no-ink red, under the field, linked via aria-describedby + aria-invalid="true". Errors clear on input.
SubmitFull-width green button (.buy vocabulary): Subscribe — send my confirmation ▸. Disabled while either field is invalid/empty and during submit. Submitting state swaps the label to Sending confirmation… and disables the whole form.
Privacy blockLocked copy string (§7) at the form's foot, small, --ink-soft, with the GitHub source link as the trust anchor.

6 · The two Telegram cards — exact specs

Both cards are sent with Telegram HTML parse_mode — escaping only & < > beats escaping MarkdownV2's 18 special characters, which appear in every dotted plan name and price (HKG.AS3.Pro.TINY, $39.90). link_preview_options.is_disabled = true so the first cart link never unfurls. Anchor text keeps mobile lines short — never raw URLs.

Digest card (one per subscriber per restock-event window)

🟢 Restock — 3 of your plans are IN STOCK

• HKG.AS3.Pro.TINY — $39.90/mo — Hong Kong · Buy now
• HKG.AS3.Pro.MICRO — $179.90/mo — Hong Kong · Buy now
• TYO.AS3.Pro.MICRO — $189.90/mo — Tokyo · Buy now
as of 14:32 UTC · data can lag ~5 min
manage: vps-stock.daichenlab.com/?manage=1
14:32 
<b>🟢 Restock — {n} of your plans {is/are} IN STOCK</b>

• <b>{escHtml(plan.name)}</b> — {price}/{period} — {city} · <a href="{deepLink}">Buy now</a>
{…one line per newly-IN subscribed plan, snapshot encounter order…}

as of {HH:MM} UTC · data can lag ~5 min
manage: vps-stock.daichenlab.com/?manage=1
  • Header counts; singular form at n = 1: “1 of your plans is IN STOCK”.
  • Only the subscriber's own plans appear — the fired-edge set intersected with their plan list; empty intersection = no message at all.
  • “as of” is the edge time, not the send time — honest even if the send retried for minutes.
  • Truncation: beyond 25 plan lines, cut and append +{k} more — see the board (plain text, board link is already in the footer) — comfortably inside Telegram's 4096-char cap.
  • Footer carries the two promises: the ~5-min honesty note and the manage deep link. No other links, no branding noise.

Confirmation card (on subscribe; “updated” variant on manage-update)

✅ Subscribed — VPS Stock Watch

You'll get ONE digest card here when any of your 3 plans restock:

• HKG.AS3.Pro.MICRO — Hong Kong

checks run ~every 5 min · a plan re-alerts only after it goes out of stock again

You'll get ONE digest card here when any of your {n} plans restock:

• {escHtml(plan.name)} — {city}
{…every subscribed plan; same 25-line truncation rule…}

  • No prices, no buy links — nothing is in stock yet; the card is a receipt, not a pitch. Same visual grammar as the digest so the digest is instantly recognizable later.
  • Doubles as the delivery proof (PRD D3): the page's success copy points at it — “Check your Telegram — your confirmation card just arrived.”
  • The manage-update variant re-uses the layout with header 🔄 Subscription updated; unsubscribe sends nothing (the user is standing in the UI that just confirmed it).

8 · Freshness ladder recalibration — the 5-minute surface

SurfaceRound 1 (60 s cadence)Round 2 (300 s cadence)
Pill: fresh → aging> 120 s (hardcoded)> 2 × cadence = 10 min — derived from cadenceSec in the envelope (D7); null-safe fallback to 120 s for old envelopes
Pill: aging → stale (banner + dim)> 300 s (hardcoded)> 4 × cadence = 20 min; same fallback (300 s)
sr-only tier announcements“under 2 minutes”, “more than 2”, “more than 5”“updated under 10 minutes ago” / “more than 10 minutes ago” / “offline for more than 20 minutes” — derived from the same thresholds, never restated as literals
Explainer (.prem-note)“reads each provider's store every ~60 s”“reads each provider's store about every 5 minutes from a real browser session … The pill above tells you exactly how old the data is.”
Footer “How this works”“polls … every ~60 s … can lag by a minute or two”“polls the providers' order pages about every 5 minutes … Status can lag by a few minutes — the ‘updated’ pill is the honest age of what you see.” + new sentence: “Restock alerts are sent by your own Telegram bot — this site can't message anyone a bot didn't already talk to.”
COPY.noneIn empty state“the board refreshes about every minute.”“Nothing in stock right now — the board checks about every 5 minutes. Subscribe to get a Telegram card the moment a plan comes back.”
<meta description> / title“updated about every minute”“checked about every 5 minutes by an independent watcher”
Stale banner copyUnchanged — “Data may be stale — watcher offline · last update {age}” is cadence-neutral by design

Why 2×/4×: one missed poll must never look like an incident (2× tolerates a full missed cycle + jitter), while a genuinely dead watcher still surfaces within one coffee break (20 min). The multipliers ship as named constants; the numbers above are derived examples, not literals to hardcode — QA's boundary tests run at both 60 s and 300 s cadences.

10 · Accessibility

  • Dialog semantics — panel root: role="dialog" aria-modal="true" aria-labelledby="subTitle". Overlay + panel are siblings at body level; the board container gets inert while open (fallback: aria-hidden + focus trap).
  • Focus — on open, focus moves to the panel's first focusable (the active tab). Tab/Shift-Tab cycle inside the panel (trap). Esc closes (except mid-SUBMITTING). On close, focus returns to the invoking element — the CTA pill, that card's bell, or body for the deep link.
  • Tabs — role="tablist"/"tab"/"tabpanel", aria-selected, roving tabindex, Left/Right arrow keys switch.
  • Picker — native <details>/<summary> accordions (keyboard for free); each row is a <label> wrapping its checkbox (whole 44 px row is the hit target); per-DC “select all” is a real tri-state checkbox (indeterminate when partial). The running count lives in the tray AND in an sr-only role="status" that announces on change, debounced 800 ms (“4 plans selected”).
  • Form — every input has a visible <label for>; errors are text nodes referenced by aria-describedby with aria-invalid="true" on the field; the async submit status (sending / success / error) is a single role="status" aria-live="polite" line — reusing the round-1 “announce transitions, never ticks” discipline.
  • Buttons — bells and copy-chips are real <button type="button"> with aria-label carrying the plan name (“Notify me when HKG.AS3.Pro.MICRO restocks”); all ≥ 40 px hit area; shared :focus-visible outline (2 px blue, offset 2).
  • Reduced motion — no drawer slide, no tray transitions; states remain legible via color + copy (as round 1).
  • Contrast — all new pairs reuse shipped tokens: --accent-ink on --blue-soft, --no-ink on --no-soft, --ok-ink on --ok-soft, white on --ok ≥ 4.5:1 (large/bold).

12 · Interaction contract for Frontend

TopicContract
Module splitNew public/js/subscribe.js module owns the panel, picker, forms, and all POSTs. board.js's poll loop stays byte-untouched except: cardHTML renders the bell, paint() keeps working (bells re-render with each snapshot — the open panel is never re-rendered by paint(); picker data refreshes only when the panel opens), and the header comment's “no POST anywhere” claim moves to “all mutations live in subscribe.js” (see §13).
Open/close APIopenPanel({ tab: 'subscribe'|'manage', preselect?: planId }) / closePanel(). Entry A → {tab:'subscribe'}; Entry B → {tab:'subscribe', preselect}; ?manage=1 → {tab:'manage'} after first paint.
Ids/classes#alertsCta · .notify[data-plan-id] (delegated click on the board container) · #subOverlay · #subPanel · #subTitle · #tabSubscribe #tabManage · #planPicker · .pick-dc (details) · .pick-all · .pick[data-plan-id] · #pickCount · #pickTray · #toStep2 #backTo1 · #fldToken #fldChat #findChat · #subSubmit · #subStatus · #subSuccess · #mgToken #mgChat #mgLoad #mgUpdate #mgUnsub.
Picker dataBuilt from the latest snapshot at open time: DC order and plan order = snapshot encounter order (as the wall); row = checkbox + name + price + status word. Plan identity = the snapshot's plan.id. If the panel is open when a snapshot arrives, the picker is NOT rebuilt (no selection loss); counts refresh next open.
API — subscribePOST /api/subscribe body {token, chatId, planIds[]} → 200 {ok, plans, merged} · 400 malformed shape/ids · 413 body cap · 422 {reason:'token_rejected'|'chat_not_found'} (confirmation send failed — nothing stored) · 429 + Retry-After · 5xx. Merged=true renders the mergedBody success variant.
API — chat-id discoveryPOST /api/chatid body {token} → 200 {chatId} · 422 {reason:'token_rejected'|'no_chat'} · 429. Fills #fldChat on success; no_chat shows errDiscover.
API — managePOST /api/subscription/lookup {token, chatId} → 200 {planIds[]} · 404 uniform (one body, no detail) · 429.
POST /api/subscription/update {token, chatId, planIds[]} → 200 {ok, plans} (n ≥ 1) · 404 · 422 · 429.
POST /api/subscription/delete {token, chatId} → 200 {ok} · 404 · 429.
Transport rulesEverything is fetch POST with Content-Type: application/json, ~10 s AbortSignal.timeout. Token and chat id ride only in POST bodies — never in URLs, query strings, or GET responses. No form ever native-submits: <form> is used for Enter-key semantics with preventDefault, so CSP form-action 'none' and connect-src 'self' stay exactly as shipped.
Buttons during flightAny in-flight request disables the whole active form and shows sending-class copy in #subStatus; concurrent submits are impossible by construction.
Freshness (D7)freshTier(ageMs, cadenceMs) — thresholds 2×/4× cadence, fallback 120 s/300 s when the envelope lacks cadenceSec. TIER_MSG strings and all §8 copy derive from the served cadence (rounded to whole minutes), computed once per envelope, not per tick.
VerificationPlaywright screenshots at 320/390/768/1440 for: panel step 1 + step 2, success, each error, manage loaded + not-found, and the board with bells; zero horizontal overflow everywhere; reduced-motion pass; keyboard-only subscribe run.

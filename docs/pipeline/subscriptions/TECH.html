3 · Data model — /data/board.db beside snapshot.json B1

Decisions this diagram pins

  • Tokens are send-credentials, so they cannot be one-way-hashed. At rest: AES-256-GCM (node:crypto), fresh 12-byte IV per write, tag appended to the ciphertext, key from TOKEN_KEY. Auth and duplicate detection never decrypt — they resolve lookup_hash = sha256(token + ':' + chatId). chat_id stays plaintext by consult recommendation: a numeric id alone can message no one; encrypting it would buy nothing while complicating lookup debugging.
  • Seeding is baseline-only (green box): the first push against a fresh DB — and every plan later added to the watchlist — creates its plan_edges row from the observed status without ever running the diff. An unknown first observation creates no row at all: the plan holds until its first confirmed status. QA's "fresh store + push with IN plans → zero sends" test is the gate.
  • Cap = 500 subscriber rows (orchestrator-pinned). Checked on the insert path only — merges/updates never count. Past cap → 429 {reason:'cap', error:'subscriber cap reached'} with no Retry-After (it is not a time-window limit — the client keys on reason:'cap' and renders the dedicated errCap copy, never a countdown). Disabled rows hold their slot until unsubscribed — no auto-purge (YAGNI at this scale; a manual sqlite3 delete is the escape hatch); at 500-row scale, table scans and the 256 MB mem limit are non-issues.
  • Key rotation is survivable: a GCM auth failure at send time disables the row (throttled warn, row id only). A later manage-update re-encrypts with the current key and — because update re-proves delivery (§B3) — re-enables it.

4 · Edge → digest pipeline — push-time diff, 75 s window, crash-safe dispatch B2

Semantics worth stating in prose

  • Level-triggered, gap-tolerant: the diff compares each accepted push against persisted per-plan state, not against the previous push — a watcher gap or server restart spanning an OUT→IN still fires exactly once. Flaps hidden entirely inside a gap are an accepted loss at 5-min cadence (consult, adopted).
  • Window scope honesty: family polls stagger across the full 300 s cycle, so a mass restock spread wider than 75 s produces a second window and a second card. That satisfies D4 as written — ONE card per restock event window, never one per plan — and 75 s still coalesces the common case (edges inside one family push burst plus adjacent families). Widening toward 300 s would trade up to 5 extra minutes of alert latency for rarer split cards; the PM already priced latency at ~5 min, so we spend none of it here.
  • Re-fire dedup during the window: because disarm persists only at window close, a second push seeing the plan still IN re-fires into the Set — where dedup absorbs it. No timing hole.
  • Disable accounting: fail_count counts consecutive exhausted dispatches (each = 3 raw attempts); it resets to 0 on any success. 5 consecutive exhausted dispatches → disabled=1. Telegram 401/403 short-circuits — one definitive rejection is enough (R6: a revoked token stops being retried).
  • Disabled rows receive nothing — the window-close transaction skips them at enqueue time (WHERE disabled=0), not at send time, so the queue never accumulates dead work.

Argued pins (adopted defaults + reconciled contracts, made precise)

  • Body shape checks match the client's relaxed formats: token /^\d{8,12}:[A-Za-z0-9_-]{30,}$/, chatId /^-?\d+$/, both on trimmed strings — the server is never stricter than the client (a stricter server regex would 400 tokens the UI accepts, with no user recourse); D3's confirmation send stays the real authority either way.
  • chatId is a string end-to-end: validated, stored (chat_id TEXT), fed to lookup_hash, and echoed as a string — the server never coerces it to a number, mirroring the client's own int64-safety rule.
  • Subscribe merges by union; update replaces. A duplicate subscribe expresses additive intent from the picker ("also these") and is never an error (DESIGN §4) — union, merged:true, confirmation card lists the full merged set. Manage-update is the editing surface whose picker shows the entire stored list pre-checked — what you see is what you save: replace.
  • The 🔄 update receipt is D3-gated, not best-effort: receipt sends first with the new plan list; only a Telegram ok persists the update (failure → 422, nothing changed). This keeps one invariant everywhere — every stored row proved deliverability at its last mutation — and gives disabled/key-rotated rows a self-service heal path (§B1). The orchestrator default only fixed that the receipt sends; ordering it as the gate is the strictly safer refinement.

Dialog lifecycle — pinned rules

  • Open — openPanel({tab:'subscribe'|'manage', preselect?}): remember document.activeElement as invoker → show overlay + panel → document.body.style.overflow='hidden' → #pageShell.setAttribute('inert','') (plus aria-hidden="true" as belt-and-braces for pre-inert engines; the manual trap below carries old browsers) → build/refresh picker → select tab → focus the active tab (first focusable).
  • Focus trap — one panel-scoped keydown handler: Tab/Shift-Tab cycle across visible enabled focusables; Esc closes except while busy (SUBMITTING/LOOKING/UPDATING ignore Esc until the request settles — bounded by the 12 s client timeout, so “stuck open” is impossible).
  • Close — Esc / overlay click / #subClose / #subDone / #mgClose: hide both, restore body scroll, remove inert, return focus to the invoker (fallback body for the deep link, or if the invoking bell was re-rendered away).
  • Draft memory — module-level state only: selection Set<planId>, current step, field values (they live in the DOM inputs). Closing hides, never resets; reopening resumes. Nothing ever touches localStorage/sessionStorage/cookies — a credential is not a convenience (DESIGN §4).
  • Picker freshness vs. draft — the picker is rebuilt from getLatest() on every open (pickerHTML(state, draftSet, openLoc) re-checks the draft); plan ids that vanished from the snapshot drop out of the draft silently. While the panel is open, an arriving snapshot never rebuilds the picker (no selection loss) — counts and statuses refresh on next open. paint() and the panel never touch each other's DOM.

The one api() helper — transport rules made unskippable

const API_TIMEOUT_MS = 12_000; // outlives the server's own 10 s Telegram abort (backend §B3/§B4) — the client
                               // can never quit first and claim “nothing was saved” after a stored success
async function api(path, body) {          // → { status, reason, retryAfter, body } — NEVER throws
  try {
    const res = await fetch(path, {
      method: 'POST',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify(body),
      cache: 'no-store',
      signal: AbortSignal.timeout(API_TIMEOUT_MS),
    });
    let data = null; try { data = await res.json(); } catch { /* empty/invalid body is fine */ }
    return { status: res.status, reason: data?.reason ?? null,
             retryAfter: parseRetryAfter(res.headers.get('retry-after')), body: data };
  } catch { return { status: 0, reason: null, retryAfter: null, body: null }; } // network / timeout
}
  • Credentials ride only in POST bodies — never URLs, query strings, headers, or GET responses. token and chatId are trimmed strings end-to-end; the client never calls Number() on a chat id (Telegram ids are int64 — large group ids would lose precision past 253). The backend stores/compares them as strings to match.
  • Cleared after success, never persisted — success empties the credential inputs; there is no storage API call anywhere in the bundle (QA greps for localStorage|sessionStorage|document.cookie|indexedDB → zero hits).
  • Never logged — no console.* in the module; the token appears in the DOM only as an input's value; error bodies are read for reason only and never interpolated into the page (all user-visible strings come from SUB_COPY).
  • One request in flight — a module-level busy flag disables the active form + tray primary before any call; concurrent submits are impossible by construction.
  • No auto-retry — the user retries. A timed-out or failed subscribe retried by the user is safe because a duplicate {token, chatId} merges server-side (200 merged:true) — the “nothing was saved” copy plus merge semantics make retry the correct universal instruction.
  • Status-only routing — status 0 and every unrecognized status ≥ 400 map to the errServer path; unknown reason strings inside a 422 fall back to errServer too. Forward-compatible by default.
  • Retry-After — integer seconds parsed; absent/unparseable → 60. Exception: a 429 whose body carries reason:'cap' (the 500-row store cap) bypasses the countdown entirely → errCap, form re-enabled — waiting 60 s at a full board would be a lie. The client stores a deadline (Date.now() + s×1000) and renders errRate with the remaining seconds at 1 Hz; the countdown survives panel close/reopen (recomputed from the deadline, not a decrementing counter) and the timer runs only while the banner is visible.

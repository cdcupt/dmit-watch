2 · E2E method & seams

The E2E loop ran the real board-server/server.js module over real loopback HTTP on 127.0.0.1:8299, with fresh PUSH_TOKEN + TOKEN_KEY (openssl rand -hex 32) and SUB_DB_FILE/SNAPSHOT_FILE in a scratch tmpdir. All client traffic was plain curl. Telegram was a local fake HTTP server that records every /bot<token>/<method> request and answers {ok:true} (switchable to 401 / hang for the failure stages).

  • Egress seam: the base is a factory option, not an env var — createBoardServer({ telegramFetch }) is the documented egress seam (TECH §Q0 ①; server.js:201). The launcher passed a fetch that rewrites https://api.telegram.org → the fake's base; every outbound call (confirmation, receipt, digest dispatch, getUpdates) rode it.
  • Digest window seam: windowMs is the documented knob (TECH §Q0 ③; default 75 000 ms, "tests ~15"). The E2E used windowMs = 5 000 ms — long enough to be observed as a real wall-clock window over the wire, short enough to run the seven stages; each stage waited 7 s (> window + dispatch) before asserting.
  • Fixtures: snapshots adapted from test/fixtures/board-snapshot.json. Plan X = lax-as3-tiny (LAX.AS3.Pro.TINY, $10.90/mo, Los Angeles — OUT in snapshot A, IN in snapshot B). Plan Y = hkg-an5-mini (HKG.AN5.Pro.MINI — OUT throughout). Envelopes carried cadenceSec: 300.
  • Credentials: sentinel-shaped fakes per the §Q0 ⑤ convention (token embeds TESTSENTINEL); shown as <redacted> below, as are all chat ids.
